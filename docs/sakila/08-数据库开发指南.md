## 1. 核心概念速览

在开始之前，先理解几个本项目用到的关键术语：

| 术语 | 通俗解释 |
|------|----------|
| **数据库 (Database)** | 存放所有数据的总容器，本项目叫 `sakila` |
| **模式 (Schema)** | 数据库内的命名空间，本项目用 PostgreSQL 默认的 `public` 模式 |
| **表 (Table)** | 存储具体数据的二维结构，类似 Excel 表格 |
| **主键 (Primary Key)** | 每行数据的唯一标识，例如 `actor_id` |
| **外键 (Foreign Key)** | 关联另一张表的字段，确保引用数据存在 |
| **触发器 (Trigger)** | 数据发生变化时自动执行的代码，本项目用于自动更新 `last_update` 字段 |
| **视图 (View)** | 基于查询结果的虚拟表，不存储真实数据 |
| **连接字符串** | 告诉程序"去哪个服务器、用什么账号、连哪个数据库"的字符串 |

## 2. 数据库结构总览

### 2.1 表关系图

```
country ──< city ──< address ──< customer ──< rental ──< payment
                        │                         │
                        ├──< staff ──< store       │
                        └──< store                 │
                                                   │
language ──< film ──< inventory ──────────────────┘
              │
              ├──< film_actor >──< actor
              ├──< film_category >──< category
              └──< film_text  (同步冗余表)
```

> `──<` 表示"一对多"关系，例如一个 city 包含多个 address。

### 2.2 表清单及用途

| 表名 | 用途 |
|------|------|
| `actor` | 演员信息（姓名） |
| `film` | 电影信息（标题、时长、评级、租金等） |
| `film_actor` | 电影与演员的多对多关联 |
| `film_category` | 电影与分类的多对多关联 |
| `film_text` | 电影标题/简介的冗余表，由触发器自动同步 |
| `category` | 电影分类（Action、Comedy 等） |
| `language` | 语言表 |
| `inventory` | 库存（哪个门店有哪部电影的哪张碟） |
| `rental` | 租借记录 |
| `payment` | 付款记录 |
| `customer` | 客户信息 |
| `staff` | 员工信息 |
| `store` | 门店信息 |
| `address` | 地址信息 |
| `city` | 城市信息 |
| `country` | 国家信息 |

### 2.3 自定义类型

```sql
-- 电影评级枚举，只能是这 5 个值之一
TYPE RATING AS ENUM ('G', 'PG', 'PG-13', 'R', 'NC-17')

-- 年份域，限制在 1901~2155 之间
DOMAIN YEARONLY AS INT
```

### 2.4 视图（只读，不含真实数据）

| 视图名 | 用途 |
|--------|------|
| `customer_list` | 客户完整信息（含地址、城市、国家） |
| `film_list` | 电影完整信息（含分类、演员列表） |
| `nicer_but_slower_film_list` | 同上，演员姓名格式化为首字母大写 |
| `staff_list` | 员工完整信息 |
| `sales_by_store` | 各门店销售额汇总 |
| `sales_by_film_category` | 各分类销售额汇总 |
| `actor_info` | 演员参演的所有电影（按分类组织） |

## 3. 如何创建数据库

### 3.1 前置条件

确保 PostgreSQL 已安装并在运行，默认账号为：

```
用户名: postgres
密码:   LuoHao@123
主机:   localhost
端口:   5432
```

### 3.2 一键初始化（推荐）

```bash
# 进入项目目录，激活虚拟环境
source venv/bin/activate

# 运行初始化脚本（完整流程：建库 → 建表 → 导入数据）
python init_db.py
```

脚本会依次执行 4 个步骤：

```
Step 1: 检查 PostgreSQL 连接
Step 2: 创建 sakila 数据库（已存在则跳过）
Step 3: 检测到 SQL 文件 → 跳过 ORM 建表（由 Step 4 处理）
Step 4: 执行 data/sakila-schema.sql 建表 → 执行 data/sakila-data.sql 导入数据
```

### 3.3 初始化脚本的可选参数

```bash
# 跳过连接检查（已知连接正常时用）
python init_db.py --skip-check

# 跳过建库（数据库已存在时用）
python init_db.py --skip-create

# 跳过建表
python init_db.py --skip-tables

# 只建库+建表，不导入数据
python init_db.py --skip-data

# 组合使用：只重新导入数据（库和表都已存在）
python init_db.py --skip-check --skip-create --skip-tables
```

### 3.4 手动创建（了解原理）

如果你想了解底层发生了什么，可以手动执行：

```bash
# 1. 登录 PostgreSQL
psql -U postgres -h localhost

# 2. 在 psql 命令行中创建数据库
CREATE DATABASE sakila;

# 3. 连接到 sakila 数据库
\c sakila

# 4. 执行 schema 文件（建表、触发器、视图、函数）
\i /path/to/python-in-practice/data/sakila-schema.sql

# 5. 执行 data 文件（导入 1000 部电影、200 名演员等样本数据）
\i /path/to/python-in-practice/data/sakila-data.sql

# 6. 退出
\q
```

### 3.5 验证创建成功

```bash
psql -U postgres -h localhost -d sakila -c "\dt"
```

应该看到 16 张表：actor, address, category, city, country, customer, film, film_actor, film_category, film_text, inventory, language, payment, rental, staff, store。

## 4. 如何连接数据库

### 4.1 连接字符串格式

```
postgresql://用户名:密码@主机:端口/数据库名
```

本项目的连接字符串（密码中 `@` 需要 URL 编码为 `%40`）：

```
# 同步连接（用于脚本、初始化）
postgresql://postgres:LuoHao%40123@localhost:5432/sakila

# 异步连接（用于 FastAPI 接口，asyncpg 驱动）
postgresql+asyncpg://postgres:LuoHao%40123@localhost:5432/sakila
```

这两个值在 `config/settings.py` 中配置，也可以通过 `.env` 文件覆盖。

### 4.2 项目中的两套引擎

`config/database.py` 维护了两套数据库引擎：

```
同步引擎 (engine)
├── 驱动: psycopg2
├── 用途: init_db.py 脚本、数据库维护操作
├── search_path: public
└── 使用方式: with Session(engine) as session: ...

异步引擎 (async_engine)
├── 驱动: asyncpg
├── 用途: FastAPI 所有 API 接口
├── search_path: public
└── 使用方式: async with AsyncSessionLocal() as session: ...
```

**为什么要两套？** FastAPI 是异步框架，异步引擎性能更高；但普通脚本不支持异步，需要同步引擎。

### 4.3 在 FastAPI 接口中获取数据库会话

FastAPI 通过依赖注入提供数据库会话，你不需要手动管理连接：

```python
from config.database import get_async_db
from sqlalchemy.ext.asyncio import AsyncSession
from fastapi import Depends

@router.get("/actors")
async def list_actors(db: AsyncSession = Depends(get_async_db)):
    # db 是自动注入的数据库会话，用完后自动关闭
    result = await db.execute(select(Actor))
    return result.scalars().all()
```

### 4.4 用 psql 命令行连接（调试用）

```bash
# 基本连接
psql -U postgres -h localhost -d sakila

# 常用 psql 命令
\dt          # 列出所有表
\dv          # 列出所有视图
\d actor     # 查看 actor 表结构
\df          # 列出所有函数
\dp          # 列出权限

# 执行 SQL
SELECT COUNT(*) FROM actor;
SELECT * FROM film LIMIT 5;
\q           # 退出
```

### 4.5 用 Python 脚本直接连接（调试用）

```python
import psycopg2

conn = psycopg2.connect(
    host="localhost",
    port=5432,
    user="postgres",
    password="LuoHao@123",  # 注意：这里不需要 URL 编码
    database="sakila"
)
cursor = conn.cursor()
cursor.execute("SELECT COUNT(*) FROM actor")
print(cursor.fetchone())  # (200,)
conn.close()
```

## 5. 日常维护操作

### 5.1 重置数据库（清空所有数据，重新导入）

```bash
# 重新运行初始化脚本即可
# load_sample_data() 会先 DROP SCHEMA public CASCADE，然后重建
python init_db.py --skip-check --skip-create
```

> **原理**：`load_sample_data()` 执行的第一步是 `DROP SCHEMA public CASCADE`，这会删除 public 模式下的所有表、视图、函数、类型，然后重新创建空的 public 模式，再执行 SQL 文件。

### 5.2 只重置数据（保留表结构）

```bash
psql -U postgres -h localhost -d sakila

-- 清空所有数据（顺序很重要，先删有外键依赖的子表）
TRUNCATE payment, rental, inventory, customer,
         staff, store, film_actor, film_category,
         film_text, film, language, category,
         address, city, country, actor
CASCADE;

-- 重新导入数据
\i /path/to/python-in-practice/data/sakila-data.sql
```

### 5.3 备份数据库

```bash
# 备份完整数据库（包含结构+数据）
pg_dump -U postgres -h localhost sakila > backup_$(date +%Y%m%d).sql

# 只备份数据（不含结构）
pg_dump -U postgres -h localhost --data-only sakila > data_backup.sql

# 只备份结构（不含数据）
pg_dump -U postgres -h localhost --schema-only sakila > schema_backup.sql
```

### 5.4 恢复数据库

```bash
# 从备份恢复（先确保数据库存在且为空）
psql -U postgres -h localhost -d sakila < backup_20240101.sql
```

### 5.5 查看数据库状态

```bash
# 查看各表数据量
psql -U postgres -h localhost -d sakila -c "
SELECT
    relname AS 表名,
    n_live_tup AS 行数
FROM pg_stat_user_tables
ORDER BY n_live_tup DESC;"
```

| payment | 14596 |
| film_actor | 5462 |
| inventory | 4581 |
| film | 1000 |

### 5.6 修改表结构（添加列）

```sql
-- 示例：给 customer 表添加一个手机号列

-- 给新列添加注释
```

## 6. 常见问题与注意事项

### 6.1 密码含特殊字符时的 URL 编码

本项目密码 `LuoHao@123` 中的 `@` 符号在 URL 格式连接字符串里有特殊含义（分隔用户名和主机），必须编码为 `%40`：

```python
# ✅ 正确：URL 格式（用于 SQLAlchemy）
"postgresql://postgres:LuoHao%40123@localhost:5432/sakila"

# ✅ 正确：字典格式（用于 psycopg2）
psycopg2.connect(user="postgres", password="LuoHao@123", ...)

# ❌ 错误：URL 格式忘记编码
"postgresql://postgres:LuoHao@123@localhost:5432/sakila"  # 解析失败
```

常见特殊字符编码对照：

| 字符 | URL 编码 |
|------|----------|
| `@` | `%40` |
| `#` | `%23` |
| `$` | `%24` |
| `%` | `%25` |
| `空格` | `%20` |

### 6.2 外键约束导致删除失败

Sakila 的表之间有大量外键约束（`ON DELETE RESTRICT`），删除父表数据前必须先删除子表数据：

```sql
-- ❌ 错误：直接删除 customer，rental 还引用它
DELETE FROM customer WHERE customer_id = 1;
-- ERROR: update or delete on table "customer" violates foreign key constraint

-- ✅ 正确：先删子表数据
DELETE FROM payment WHERE customer_id = 1;
DELETE FROM rental WHERE customer_id = 1;
DELETE FROM customer WHERE customer_id = 1;
```

依赖关系速查（删除时从左到右操作）：

```
country → city → address → customer → rental → payment
                          → staff → store
language → film → inventory → rental
```

### 6.3 `GENERATED ALWAYS AS IDENTITY` 主键不能手动插入

本项目所有主键都使用 `GENERATED ALWAYS AS IDENTITY`（自增），正常插入时**不要指定主键值**：

```sql
-- ✅ 正确：不指定 actor_id，数据库自动生成
INSERT INTO actor (first_name, last_name) VALUES ('Tom', 'Hanks');

-- ❌ 错误：指定了 actor_id 会报错
INSERT INTO actor (actor_id, first_name, last_name) VALUES (999, 'Tom', 'Hanks');
-- ERROR: cannot insert into column "actor_id" (GENERATED ALWAYS)
```

初始化脚本 `sakila-data.sql` 使用了特殊语法导入历史数据：

```sql
-- OVERRIDING SYSTEM VALUE 允许在导入时指定自增列的值（仅数据迁移时使用）
INSERT INTO actor OVERRIDING SYSTEM VALUE VALUES (1, 'PENELOPE', 'GUINESS', ...);
```

### 6.4 结构变更用 Alembic，不要手动 ALTER

在开发阶段可以手动 `ALTER TABLE`，但正式项目中应使用 **Alembic** 管理数据库版本：

```bash
# 初始化 Alembic（只需一次）
alembic init alembic

# 自动生成迁移文件（对比 ORM 模型与实际数据库的差异）
alembic revision --autogenerate -m "add phone to customer"

# 执行迁移（升级到最新版本）
alembic upgrade head

# 回滚一个版本
alembic downgrade -1
```

**为什么不能手动 ALTER？** 手动修改不留记录，团队协作时别人不知道数据库改了什么，部署到新环境时也无法重现变更历史。

### 6.5 `search_path` 决定操作哪个 Schema

PostgreSQL 支持在同一数据库中创建多个 Schema（命名空间）。本项目使用 `public`（默认值），在代码中通过以下方式明确指定：

```python
# config/database.py
# 同步引擎
connect_args={"options": "-csearch_path=public"}

# 异步引擎
connect_args={"server_settings": {"search_path": "public"}}
```

如果不设置，PostgreSQL 也默认使用 `public`，但显式指定可避免歧义。

### 6.6 触发器自动维护 `last_update`

所有表都有 `last_update` 字段，由触发器 `set_last_update_func()` 在每次 UPDATE 时自动更新：

```sql
-- 执行 UPDATE 时不需要手动设置 last_update
UPDATE actor SET first_name = 'PENNY' WHERE actor_id = 1;
-- last_update 字段会被触发器自动更新为当前时间
```

> **重要**：触发器函数必须包含 `RETURN NEW;`，否则 BEFORE UPDATE 触发器会静默取消更新操作。`data/sakila-schema.sql` 已修复此问题。

### 6.7 重新初始化会清空所有数据

`python init_db.py` 在执行 SQL 文件之前会运行：

```sql
DROP SCHEMA public CASCADE;   -- 删除 public 下的所有对象
CREATE SCHEMA public;          -- 重建空的 public 模式
```

**这是破坏性操作**，会删除 `public` 模式下的所有表和数据。如果你在数据库中有自定义数据，运行 `init_db.py` 前请先备份：

```bash
pg_dump -U postgres -h localhost sakila > my_backup.sql
python init_db.py
```

### 6.8 连接池与连接泄漏

SQLAlchemy 使用连接池管理数据库连接，正确使用方式：

```python
# ✅ 正确：使用 with 语句，自动归还连接到池
with Session(engine) as session:
    result = session.execute(select(Actor))

# ✅ 正确：FastAPI 中通过依赖注入，框架自动管理
async def get_actors(db: AsyncSession = Depends(get_async_db)):
    ...

# ❌ 错误：手动创建 session 但忘记关闭
session = SessionLocal()
result = session.execute(select(Actor))
# 如果这里发生异常，连接永远不会归还
```
