## 三阶段工作流配置
### 1. 研究阶段（RESEARCH）
每个任务开始前必须执行：
- 使用 Glob 搜索相关文件：`*.py`
- 使用 Grep 搜索关键词：函数名、类名、相关业务术语
- 检查现有代码库中的类似实现
- 阅读相关模块的注释
- 理解项目架构和依赖关系

不确定时必须联网搜索：
- 异步 SQLAlchemy 最佳实践
- FastAPI 异步路由写法
- 错误消息的解决方案

### 2. 开发阶段（DEVELOPMENT）
严格遵循项目现有架构与编码规范
- 按「模型→Pydantic→服务→接口→测试」标准流程开发
- 所有代码必须符合下方**编码标准**与**质量红线**
- 开发过程中实时运行测试
- 完成后执行 black + isort

### 3. 验收阶段（ACCEPTANCE）
提交代码前必须完成：
- 执行全量测试用例，确保通过率 100%
- 检查代码是否符合所有质量红线
- 确认无硬编码、无敏感信息
- 注释完整，逻辑清晰

## 质量红线（绝对禁止清单）
以下行为绝不允许，没有例外：

### 代码质量
- 绝不提交未通过测试的代码
- 绝不使用 TODO/FIXME/HACK 作为最终代码
- 绝不跳过错误处理
- 绝不吞掉异常（空 except 块）
- 绝不使用魔法数字（必须定义常量）

### Python 项目强制规则
- 绝不编写同步数据库操作（必须用 async/await）
- 绝不在路由函数中直接写 SQL / ORM 逻辑（必须走服务层）
- 绝不跳过 Pydantic 数据校验
- 绝不直接创建数据库会话（必须使用依赖注入 get_async_db）

### 安全相关
- 绝不硬编码密钥、密码、Token
- 绝不在代码中包含敏感信息
- 绝不信任未验证的用户输入

## 编码标准
### 命名规范
- 文件/变量/函数：snake_case
- 类：PascalCase
- 常量：SCREAMING_SNAKE_CASE
- 私有成员：_leading_underscore
- 布尔值：is/has/can/should 前缀

### 函数规范
- 单个函数不超过 30 行
- 参数不超过 4 个，超过使用 dataclass
- 必须添加类型注解
- 一个函数只做一件事

### 异步规范
- 所有数据库操作必须使用 async def + await
- 路由函数必须是异步函数 async def
- 服务层必须全部异步

### 错误处理
- 绝不使用 bare except
- 捕获具体异常类型
- 使用 Optional 处理可能为 None 的值

### 代码组织
- 导入顺序：标准库 → 第三方库 → 本地模块
- 公开函数必须有 Docstring
- 遵循 PEP 8 规范
